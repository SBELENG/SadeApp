Stack Tecnológico

La arquitectura propuesta es una Single Page Application con backend REST desacoplado. Esta separación permite escalar el motor de cálculo de manera independiente al frontend y facilita futuras integraciones con sistemas hospitalarios (HIS/HER).

06

Algoritmo de Cálculo — Metodología Balderas Pedrero

El motor de cálculo ejecuta 5 pasos secuenciales. Todos los resultados se redondean al entero superior (Math.ceil) para garantizar cobertura. La función reside en utils/dotacion.calc.ts y es ejecutada tanto en el cliente (preview) como en el servidor (validación definitiva).

Paso 1 — Personal Base (P)

Determina la dotación teórica ideal para cubrir 24 horas continuas de servicio.

P = (I × C) / J Personal Base
I = índice cuidado C = camas   J = 8hs

Paso 2 — Ausentismo Previsible (B)

Calcula el colchón de personal necesario para cubrir descansos, vacaciones y licencias. El coeficiente 0.41 (41%) está estandarizado por la Guía Técnica y es constante.

B = P × 0.41

Paso 3 — Dotación Total Requerida (Z)

Z = P + B

Paso 4 — Distribución por Turno (Q)

Segmentación de Z según franjas horarias. La sumatoria de los cuatro tramos debe ser 100%.

Q₁ = Z × 0.35 Mañana (H1)
Q₂ = Z × 0.25 Tarde (H2)
Q₃ = Z × 0.20 Noche (H3)
Qf = Z × 0.20 Francos
✗ Validación crítica: El sistema debe verificar que Q₁ + Q₂ + Q₃ + Qf = Z (100%). Si la suma difiere por error de redondeo, se ajusta Qf en ±1. Un error superior al 1% debe lanzar excepción.

Paso 5 — Composición Profesional (S / V)

Aplicado por turno sobre cada Q, según el nivel de complejidad configurado.

2° Nivel S = Q × 0.70  |  V = Q × 0.30
3° Nivel S = Q × 0.80  |  V = Q × 0.20

Implementación de referencia

export interface DotacionInput {
  I: number; C: number; J: 8;
  nivel: 'segundo' | 'tercero';
}

export function calcularDotacion(input: DotacionInput) {
  const { I, C, J, nivel } = input;

  // Paso 1-3
  const P = (I * C) / J;
  const B = P * 0.41;
  const Z = P + B;

  // Paso 4 — distribución
  const Q1 = Math.ceil(Z * 0.35); // Mañana
  const Q2 = Math.ceil(Z * 0.25); // Tarde
  const Q3 = Math.ceil(Z * 0.20); // Noche
  const Qf = Math.ceil(Z * 0.20); // Francos

  // Paso 5 — composición profesional
  const ratio = nivel === 'tercero' ? 0.80 : 0.70;
  const profesionales = Math.ceil(Z * ratio);
  const auxiliares = Math.ceil(Z * (1 - ratio));

  return { P, B, Z, Q1, Q2, Q3, Qf, profesionales, auxiliares };
}
07

Restricciones de Validación (REQ-001 a REQ-006)

Estas reglas son validadas en tiempo real en el hook useShiftValidation.ts al asignar o modificar cualquier celda. Una restricción crítica bloquea la acción; las de precaución muestran alerta pero permiten continuar con confirmación.

  • REQ-001
    Base de Francos: Si mes_dias = 28, 29 ó 30 → asignar 8 francos; si = 31 → 9 francos. Es el valor inicial del contador dinámico de cada enfermero.
  • REQ-002
    Ajuste por Feriado: Por cada feriado nacional marcado en el mes, Francos_Totales = Base_Francos + 1. La base se recalcula automáticamente al marcar/desmarcar feriados en Pantalla 1.
  • REQ-003
    Franco Compensatorio: Si la celda asignada corresponde a un día feriado Y el turno es M o T → la celda se pinta amarillo, se suma +1 al contador de francos compensatorios del enfermero, y una notificación advierte al usuario. Si el feriado cae el último día del mes, el compensatorio se traslada al mes siguiente (campo compensatorio_pendiente en la entidad Personal).
  • REQ-004
    Fatiga Circadiana — BLOQUEO: Prohibido asignar más de 3 turnos N consecutivos. Al intentar asignar el 4° N consecutivo, el sistema bloquea la acción (celda no modificable) y muestra alerta roja: "Límite de 3 turnos nocturnos consecutivos alcanzado."
  • REQ-005
    Continuidad Laboral — BLOQUEO: Máximo 5 días consecutivos de trabajo (M, T o N) sin un F. Al intentar asignar el 6° día sin franco, el sistema bloquea. Alerta roja: "Máximo de 5 días consecutivos sin franco superado."
  • REQ-006
    Descanso Inter-Jornada — PRECAUCIÓN: Mínimo 16 horas entre turno y turno. Regla específica: si día[n] = T (finaliza 22:00) y día[n+1] = M (inicia 06:00) → solo hay 8 horas de descanso → alerta naranja. El sistema no bloquea pero muestra confirmación obligatoria: "¿Confirmar turno? Viola 16hs de descanso mínimo."

Regla adicional: supervisión de Auxiliares

✗ REQ-007 (implícito): Si al validar un turno (M, T o N) todos los enfermeros asignados tienen nivel_formacion = AUXILIAR, el sistema muestra alerta roja: "El turno no tiene supervisión profesional. El nivel auxiliar requiere supervisión de un Enfermero Profesional." Esta regla surge de la Ley 24.004.
08

Sistema de Alertas — Semáforo Visual

Las celdas de la grilla cambian de color según el tipo de infracción detectada. Las alertas son acumulables (una celda puede ser naranja y amarilla simultáneamente). En ese caso se aplica el color de mayor prioridad (rojo > naranja > amarillo).

🔴 Alerta Roja — Restricción Crítica (BLOQUEANTE)

Disparadores: REQ-004 (+3 noches consecutivas), REQ-005 (+5 días sin franco), REQ-007 (turno solo con Auxiliares).

Comportamiento: La celda queda bloqueada y no puede modificarse. El sistema muestra un tooltip con la causa. El administrador puede desbloquear manualmente con contraseña de supervisor (modo override, queda logueado en auditoría).

🟠 Alerta Naranja — Precaución de Descanso

Disparador: REQ-006 — menos de 16 horas entre turno T y turno M del día siguiente.

Comportamiento: Modal de confirmación con dos opciones: cancelar la asignación o confirmar forzando (requiere confirmación explícita). Si se confirma, la celda queda marcada en naranja.

🟡 Alerta Amarilla — Compensatorio / Fin de Semana

Disparadores: Turno asignado en día feriado (REQ-003); fin de mes próximo sin fin de semana libre detectado.

Comportamiento: La celda se pinta amarillo suave. No bloquea ni pide confirmación. El contador de francos compensatorios se actualiza automáticamente.

🟢 Indicador Verde — Franco Cumplido

Disparador: El contador de francos restantes llega a 0.

Comportamiento: El contador al final de la fila cambia a verde. Indica que se han respetado todos los derechos de descanso del trabajador en ese mes.

Lógica del Contador de Francos por Fila

// Valor inicial calculado en useEffect al cargar la planilla

Exportación a PDF — Planilla Imprimible

El PDF exportado debe replicar fielmente el formato de la planilla de papel institucional (el documento de referencia provisto), incluyendo el logo de la institución en el encabezado. Se generará cliente-side usando jsPDF + html2canvas o directamente con la API de jsPDF para mayor control tipográfico.

Estructura del PDF exportado

  • PDF-01
    Encabezado: Logo de la institución (izquierda), nombre del establecimiento (derecha). Debajo: Área/Sector/Servicio — Mes — Año, en una línea horizontal con puntillado, replicando el formato original.
  • PDF-02
    Grilla: Tabla con columna "Personal" + columnas de días (1 a 31, adaptando al mes real). Celdas con letra M/T/N/F. Orientación apaisada (landscape) A4. Las celdas con alerta se imprimen con fondo sombreado gris claro (no a color, para compatibilidad con impresoras B&N).
  • PDF-03
    Pie de página: Contador de francos de cada enfermero en la última columna. Leyenda: M = Matutino, T = Vespertino, N = Nocturno, F = Franco. Firma del jefe de servicio y fecha de emisión.
  • PDF-04
    Ajuste de filas: Si hay más de 15 enfermeros, la grilla se divide en 2 páginas manteniendo el encabezado en ambas.
✓ Adopción crítica: La planilla en papel sigue siendo necesaria en la mayoría de instituciones para firma y archivo físico. El botón "Exportar PDF" es una funcionalidad prioritaria de MVP, no una mejora futura.

Riesgos y Consideraciones

⚠ Riesgo Alto — Performance de la Grilla

Una grilla de 30 enfermeros × 31 días = 930 celdas con estado reactivo puede generar re-renders costosos. Mitigación: usar React.memo en ShiftCell, useMemo en los cálculos de validación, y evaluar @tanstack/virtual para virtualización de filas si el número de personal supera los 40.

⚠ Riesgo Medio — Ambigüedad en REQ-003 (fin de mes)

Si un feriado cae el último día del mes, el franco compensatorio debe trasladarse al mes siguiente. La entidad Personal debe tener el campo compensatorio_pendiente (Int) que se suma automáticamente a la base de francos del mes siguiente al abrir esa planilla.

⚠ Riesgo Medio — Enfermero en múltiples servicios

El modelo actual asume 1 servicio por enfermero. Si la institución tiene personal compartido entre servicios, se necesitará una relación N:M entre Personal y Servicio (tabla personal_servicios). Evaluar con el cliente en Fase 1.

📌 Consideración — Roles de Usuario

Se recomienda implementar desde la Fase 1 al menos dos roles: Administrador (acceso completo, puede hacer override) y Visualizador (solo lectura, no puede modificar celdas). Un tercer rol Jefe de Servicio con acceso limitado a su propio servicio puede agregarse en una versión posterior.

✓ Fuera del Alcance del MVP

Las siguientes funcionalidades del documento de requerimientos se posponen a una versión 2.0: Módulo de BI Donabedian (indicadores de resultado clínico), integración con sistema HIS/HER externo, categorización de pacientes por UPE (Módulo de Carga Dinámica), Sistemas de Organización del Trabajo (por funciones, usuario o equipo).
